# football-betting workspace

Multi-repo workspace orchestrating three production services. This `.claude/`
governs how Claude operates here. Specs and architecture docs live in `docs/`,
tickets in `tickets/`.

## Repos

| Path             | Stack           | Role                                             |
|------------------|-----------------|--------------------------------------------------|
| `frontend/`      | Next.js + TS    | Web UI, owns SQLite schema & user/tip writes     |
| `betting-api/`   | Rust + Actix    | Read API (`/rating`, `/user/{id}`, `/game/{id}`) |
| `macht-api/`     | Rust + Tokio    | Cron importer (external football API → SQLite)   |
| `shared/db/`     | (no repo)       | Holds `database.db` and `test.db`                |

All three services share the SQLite file at `../shared/db/database.db`. Schema
authority lives in `frontend/db/schema.ts` (Drizzle). Rust services consume the
same tables — keep field names and types in lockstep.

Specs: `docs/FRONTEND_FUNKTIONS_SPEC.md`, `docs/TECH_ARCHITEKTUR.md`.

## Commit conventions

- Commits in **English**. Conventional Commits-style prefix preferred
  (`feat:`, `fix:`, `refactor:`, `docs:`, `chore:`).
- PR descriptions in **English**.
- **Never** add `Co-Authored-By: Claude`, `Generated by Claude`, links to
  claude.ai, or similar attribution lines. This project does not advertise its
  tooling — clean commit history only. The `confidentiality-check.sh` hook
  enforces this on every `git commit`.

## Pre-Commit Check

Before every commit:
- No debug flags, mock endpoints, or `TODO: remove before merge` left in
- No secrets, API tokens, or DB paths leaked into committed files
- `.env` files never committed (gitignored per repo)
- For schema changes: corresponding Rust struct in `betting-api` / `macht-api`
  updated in the same commit chain

## Coding standards

### Comments
Default to no comments. Only add when the *why* is non-obvious (hidden
invariants, workaround for a specific bug). Code that explains itself does not
need narration.

### Style fix
A `PostToolUse` hook runs the right formatter automatically after every
`Edit|Write`:
- `.rs` → `rustfmt`
- `.ts|.tsx|.js|.jsx|.astro|.json|.md|.css` → `prettier` (from nearest
  `package.json`)

### Project rules
- **Schema in lockstep** — touching `frontend/db/schema.ts` requires reviewing
  Rust structs in `betting-api/src/db/mod.rs` and `macht-api/src/api/match_client.rs`.
- **No `.unwrap()` in Rust production paths** unless preceded by a check that
  proves it cannot panic. Tests are OK.
- **No `any` in TypeScript**. Use `unknown` and narrow.
- **Server-only secrets**: TypeScript env vars must not be prefixed with
  `NEXT_PUBLIC_` unless intentionally exposed to the browser.
- **Match data is immutable from the frontend** — only `macht-api` writes to
  the `match` table. Frontend reads via Rust or directly via Drizzle.

## Code quality commands

Each repo has its own quality gate. Run the one that matches the touched files.

### `frontend/` (Next.js + TypeScript)
```bash
cd frontend
pnpm exec prettier --write <file>       # style
pnpm exec tsc --noEmit                  # lint / type-check
pnpm exec vitest run <test-path>        # tests
```

### `betting-api/` & `macht-api/` (Rust)
```bash
cd <repo>
cargo fmt -- <file>                     # style
cargo clippy -- -D warnings             # lint
cargo test                              # tests
```

### Quality gate (run in this order, per repo)
1. **Style fix** on every changed file (hook does it automatically)
2. **Type-check / clippy** — must be clean
3. **Tests** — all green

Before committing: re-run the style fix on the full diff (the hook only
catches what Claude edits — manual edits via the shell still need it).

## Workflow

### Ground rules
1. Read this `CLAUDE.md` before starting any task.
2. Work from a backlog ticket. No changes without a ticket.
3. Touch only files the ticket needs. No refactoring of unrelated code.
4. Keep diffs minimal — no formatting churn, no comment additions on
   untouched code, no opportunistic renames.
5. Cross-repo changes (e.g. schema migration) must land as a coordinated
   set of commits — list them in the ticket's `Scope` section.

### Ticket lifecycle

```
todo → in-progress → review → done
```

Tickets live in `tickets/backlog/<status>/<PREFIX>-<NUMBER>_<slug>.md`.
Template: `.claude/templates/ticket-template.md`.
Fixtures/test data for tickets: `tickets/testdata/`.

**Ticket ID prefixes** (one per repo, lets multiple terminals work in parallel
without ID collisions):

| Prefix | Repo          |
|--------|---------------|
| `FE`   | `frontend`    |
| `BA`   | `betting-api` |
| `MA`   | `macht-api`   |
| `WS`   | `workspace`   |
| `XR`   | cross-repo    |

Numbering is per-prefix and monotonic — never reuse a number.

Slash commands automate transitions:
- `/ticket-start <id>` — `todo/` → `in-progress/`
- `/ticket-review <id>` — `in-progress/` → `review/`
- `/ticket-done <id>` — `review/` → `done/` + write `tickets/results/<id>_<slug>.md`

Only one ticket per owner in `in-progress/` at a time.

### Agents

Definitions in `.claude/agents/`.

- **implementer** — implements a ticket end-to-end with minimal scope
- **reviewer** — verifies against acceptance criteria + framework risks (Rust
  panic paths, Next.js auth boundaries, schema-drift between Rust & TS)
- **tester** — on-demand helper that fills test gaps. Not a pipeline stage —
  invoked by implementer or reviewer.

### Definition of Done

A ticket reaches `done/` only when:

- [ ] Every acceptance criterion satisfied
- [ ] Quality Gate green in every touched repo
- [ ] No `.unwrap()`, `any`, or debug flags introduced
- [ ] Schema changes have matching Rust struct updates
- [ ] No `Co-Authored-By: Claude` or AI-attribution lines in commits
- [ ] Result document written to `tickets/results/<id>_<slug>.md`

### Result documents

After a ticket closes:
- Path: `tickets/results/<ticket-id>_<ticket-slug>.md`
- Content: what was done, which files changed (across all repos), test results
